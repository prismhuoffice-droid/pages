リマインド自動化

週次・月次・年次の 定期的な提出物 (達成度報告・実績報告・各種申請書など) について、未提出者を Google フォーム / スプレッドシートから自動検出し、本人にはメールで、事務局には Slack で 段階的にリマインドを送るシステム。
事務局スタッフの「毎週同じ作業」を完全自動化します。
PRISM-HU が運用・提供しています。

基本機能

  • 提出締切日にあわせて、未提出者を 自動判定 (Google フォームの回答スプレッドシートを読み取って提出済みを除外)
  • 未提出者本人に 段階的なリマインドメールを自動送信 (1日目=やわらかく、2日目=具体的に、3日目=締切直前の最終確認、など)
  • 事務局チャンネルに 進捗を Slack 通知 (本日の提出状況・未提出者数・送信結果)
  • 提出状況を ブラウザのダッシュボードで一目で確認 (誰が提出済み / 未提出か)
  • 締切日を過ぎたら自動で リマインドを打ち切り、その週分は終了処理 (週明け再開)

安全機能

  • 既存システムへの影響なし — 回答スプレッドシートは読み取り専用で参照
  • 個別送信失敗の検出と再送 — メール送信が失敗した相手は記録され、ワンクリックで再送可能
  • 緊急停止コマンド — 1 回の操作で全自動送信を即停止できる (誤送信時の安全弁)
  • ドライラン機能 — 実送信前に「誰に何が送られるか」を事前確認可能
  • 送信履歴の保持 — 同じ週内で同じ人に同じメールを二重送信しない (idempotency)

※ 全送信履歴は事務局 Slack にも通知されるため、見落としを防ぎます。

利用の流れ

1 お申し込み
下記フォームから教室の状況 (リマインドしたい報告物の種類・対象人数・提出締切の周期など) を教えてください。
2 事前打ち合わせ
PRISM-HU と教室管理者の間で、対象フォーム・対象メンバーリスト・リマインドスケジュール (どの曜日・時刻に何回送るか) を確認します。
3 セットアップ (数時間)
PRISM-HU が Google Apps Script プロジェクトを設定し、教室の Slack に通知 webhook を接続、自動実行スケジュールを登録します。
4 1 か月伴走
初月は週ごとに動作確認・文面調整。 教室の運用ルールに合わせてリマインド文言や送信タイミングを微調整します。
5 本運用
以降は完全自動。事務局スタッフはダッシュボードを開く・Slack通知を見るだけで進捗を把握できます。

よくある質問

どのような報告物に対応できますか?

Google フォームまたはスプレッドシートに「提出者の氏名」が記録されるものであれば、ほぼすべて対応可能です。例: 達成度報告・実績報告・出張伺・各種申請書・週次タスク報告など。提出有無を「人」単位で判定できることが条件です。

リマインドの文面はカスタマイズできますか?

はい、教室ごとに自由にカスタマイズできます。段階別 (1日目・2日目・締切直前など) の文面、件名、署名すべて変更可能です。打ち合わせ時に教室の文化に合わせて調整します。

送信スケジュールは自由に設定できますか?

はい。曜日・時刻・回数をフリーに設定できます。例: 週次なら「月8時 → 火8時 → 火14時 → 水8時 → 水17時 → 水18時打ち切り」のような段階運用、月次なら「締切3日前 → 1日前 → 当日朝」など。

メンバー追加・退職時の対応は?

教室管理者が設定ファイル (Apps Script 上の `Config.gs`) のメンバーリストを 1 行追加・削除するだけです。手順はマニュアルにも記載されています。複雑な設定変更は不要です。

誤って送信した場合に止められますか?

緊急停止コマンドが用意されており、教室管理者が Apps Script 上でワンクリック実行すれば全自動送信が即停止します。停止後、原因を確認してから再開できます。

情報セキュリティ的な懸念はありますか?

本システムは教室の Google Workspace アカウントの上で動作し、フォーム回答スプレッドシートを 読み取り専用で参照します。リマインドメールも教室の Gmail から送信されます。北大外のサーバを経由しません。Slack 通知は Slack 社のサーバを経由しますが、Slack を業務利用していること自体と同じ範囲です。

既存の業務フローを変える必要はありますか?

基本的にありません。既存の Google フォーム / 提出ルールはそのまま使えます。「未提出者にリマインドを送る」という事務局の手作業の部分だけが自動化されます。

北大の MS 包括契約で使える Teams ではダメですか?

技術的には Teams 対応も可能ですが、現状は Slack 専用で開発しています。理由は (1) 教室管理者だけで App 導入が完結する自由度、(2) Bot 開発エコシステムの成熟度、(3) PRISM-HU 提供の他ツール (サブスク領収書ボット・メール転送自動化) との設計統一、です。Teams は北大本部 IT 室の管理権限下にあり、Bot 導入時に学内承認が必要なケースがあるため、教室レベルで完結する Slack を採用しています。

途中で使うのをやめる場合はどうすればよいですか?

Apps Script の自動実行トリガーを削除すれば、リマインド送信は停止します。プロジェクト本体やデータの削除手順もマニュアルに記載しています。

ご利用にあたって

1. サービスの提供
リマインド自動化システムは、PRISM-HU が北海道大学医学研究院内の教室・研究室向けに提供する事務作業自動化ツールです。

2. 教室側でご用意いただくもの
本システムは教室の Google Workspace 上で動作します。Google Workspace のアカウントおよび Slack ワークスペースは教室にてご用意ください。

3. データの管理
フォーム回答データ・メンバーリスト・送信履歴はすべて教室の Google Workspace 内に保存されます。PRISM-HU はトラブル対応時にのみアクセスし、それ以外で教室データを参照することはありません。Slack 経由のデータは Slack 社のサーバを通過しますが、Slack を業務利用していること自体と同じ範囲です。

4. 教室管理者の責任
リマインドの送信内容・対象メンバーリストの正確性・スケジュールの妥当性は教室管理者の責任において管理してください。Apps Script の認証情報 (Slack webhook URL 等) は他者と共有しないようご注意ください。

5. 免責事項
PRISM-HU は本サービスの安定運用に努めますが、Google・Slack 等の外部サービスの障害・仕様変更等に起因する不具合について責任を負いかねます。重要なリマインドが届かない可能性に備え、ダッシュボードの定期確認を推奨します。

6. 利用の停止
ご利用が不要になった場合は、Apps Script の自動実行トリガーを削除してください。データの削除手順はマニュアルに記載されています。PRISM-HU 側で記録している教室情報の削除が必要な場合は、別途ご連絡ください。

7. 変更について
本ガイドラインおよびソフトウェア本体は、サービスの改善に伴い変更されることがあります。変更時は本ページにて告知します。

「この業務、自動化できない?」のご相談を歓迎します

本システムの導入には Slack の有料プラン契約が必要です。まだ準備が整っていない教室や、リマインド以外の事務作業で 「定型的だが手間がかかる」 作業がある場合も、ぜひお気軽にご相談ください。

業務内容を ヒアリングし、必要な機能を洗い出し、その教室に 最適なシステムを組み立てます。既存ツールの組み合わせで実現できる場合もあれば、新規にツールを設計する場合もあります。Slack 教育プランの導入支援・ベンダー橋渡しもあわせて対応します。

たとえば、以下のようなご相談を過去にいただいています:

  • 月末の 経費集計を Excel に転記する作業
  • 毎週同じメンバーへの 定型連絡 (会議案内・週報集約など)
  • 特定キーワードを含む メールの抽出・転送
  • Google フォーム回答の 集計・要約 → Slack 通知
  • 外部サイトの 定期チェック (公募情報・締切リスト)
  • カレンダーや タスク管理ツールとの 連携

業務内容を伺った上で、自動化の可否・難易度・想定スケジュールをご提案します。
すぐに自動化できない場合でも、業務フロー改善のヒントになる場合があります。
